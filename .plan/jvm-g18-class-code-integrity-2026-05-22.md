# JVM G18 Class-Code Integrity - 2026-05-22

## Scope

Implement generic JVM full-obfuscation anti-tamper enforcement for protected
class method bodies during class key-table initialization. The target bypass is
method-body instrumentation of protected methods while leaving the G18 host,
class initializer, and key table fields untouched.

## Status Legend

- `[ ]` not implemented or not freshly verified
- `[-]` active subtask
- `[x]` implemented, freshly validated, and committed

## Constraints

- Do not special-case any sample jar, owner, method, descriptor, password, seed,
  flag, or known generated name.
- Do not add new classes, JNI, JVMTI, fallback, skip-on-error, or original
  bytecode fallback.
- Keep checks on the existing JVM CFF/G18/key-table surface.
- A patched method body must fail closed before protected CFF, constant, string,
  or invokedynamic material remains usable.
- Use repository `./gradlew` only after explicit permission.

## Execution Rows

### [x] JG18-1: Enforce class code integrity before key-table use

- Scope: add a per-class runtime code digest check on the existing CFF/G18
  surface and run it before class key-table initialization proceeds.
- Required evidence: generated bytecode contains the integrity helper and
  `<clinit>` call before the CFF object table is installed; the digest covers
  method `Code` byte arrays and excludes only `<clinit>` to avoid self-reference
  through the expected digest constant.
- Validation command or runtime target: focused JVM CFF tests, full JVM
  obfuscation perf test, and a negative tamper test that mutates a protected
  method body after obfuscation.
- Completion criteria: original obfuscated artifacts run, a patched protected
  method body fails closed, and no fallback path returns a neutral digest.
- Evidence 2026-05-22: implemented a JVM output finalizer for CFF that injects
  per-class code digest helpers into existing classes and calls them at the
  start of `<clinit>` before the class key table is installed. The runtime
  parser hashes method `Code` byte arrays and excludes only `<clinit>` so the
  expected digest does not self-reference. The focused audit test now mutates
  the obfuscated `main` method body after generation and verifies that the
  tampered jar exits non-zero through `ExceptionInInitializerError` /
  `IllegalStateException`, while the original obfuscated jar still prints the
  expected output. Validation passed with
  `./gradlew -PbuildDir=build/validation-g18 :neko-test:test --tests
  dev.nekoobfuscator.test.ControlFlowFlatteningAlgebraicAuditTest
  --rerun-tasks`.

### [-] JG18-5: Poison-only mismatch propagation

- Scope: convert G18 order mismatch, class-code hash mismatch, and class-code
  parser/resource anomalies from explicit mismatch throws into keytable poison.
  The protected class must continue initializing with wrong key material; later
  CFF, invokedynamic, string, constant, or key-dispatch paths decide the
  observable failure naturally.
- Required evidence: runtime keytable decoding consumes both actual
  order-dependent G18 root and class-code digest poison material; generated
  verifier code has no explicit mismatch compare/throw; wrong order or patched
  method body cannot produce the correct protected result.
- Validation command or runtime target: focused JVM CFF tests with static patch
  and wrong-order direct-load regressions, structural audit for absent explicit
  mismatch throws, full JVM obfuscation performance test, and regenerated
  five-jar smoke/adversarial checks.
- Completion criteria: canonical runs still succeed, mismatches poison the
  keytable instead of throwing from the integrity/order check, no fallback or
  neutral-key recovery exists, and validation evidence is fresh.
- Evidence 2026-05-22: implemented poison-only propagation for class-code
  verifier mismatches and G18 order mismatches. The class-code verifier now
  returns a non-zero digest poison on missing class bytes, malformed parser
  state, or method-code hash mismatch; `<clinit>` stores that poison and the
  class key-table decode consumes it as part of every class-word mask. The G18
  helper now records a global loaded-class Bloom state and each protected class
  consumes the Bloom bits for active-use predecessors only, including static
  method/field use, `new`, invokedynamic static handles, and same-method
  active-use order. A wrong predecessor state changes the G18 root used to
  decode the class key table, but no explicit mismatch throw is emitted.
- Validation 2026-05-22: focused CFF audit passed with
